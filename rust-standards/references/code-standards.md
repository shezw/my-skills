# Rust Code Standards

这份文档是仓库内的 Rust 规范正式副本，供 `rust-standards` 技能直接引用；用户级 `rust.instructions.md` 负责把同类规则自动应用到 `*.rs` 文件。

## Goals

- 优先保证模块边界清晰、代码可读、行为可验证。
- 先做高内聚拆分，再做复用抽象；不要为了“看起来模块化”而过度拆文件。
- 默认使用 Rust 社区成熟实践：`rustfmt`、`clippy`、明确的文档注释、最小暴露面、可测试的错误处理。

## Project Layout And Module Boundaries

- 单个模块只承担一个清晰职责；结构体、相关 `impl`、紧耦合辅助函数默认放在同一模块，直到职责明显分叉再拆分。
- 默认模块深度不超过 3 层，例如 `crate::feature::parser::lexer`；超大型项目才允许额外层级，并且要在顶层文档说明原因。
- 优先按业务能力或领域边界拆分，不按“类型种类”机械拆成 `types.rs`、`traits.rs`、`utils.rs`。
- `main.rs` 和 `lib.rs` 保持薄入口；业务逻辑应下沉到领域模块，而不是堆积在 crate 根。
- workspace 只有在存在独立发布、清晰领域边界或编译隔离价值时才拆 crate，不要过早用多 crate 代替正常模块化。
- 小型功能优先保留在一个文件中；只有在职责分叉、测试或文档负担明显上升、或单文件持续膨胀时再拆分。
- 避免创建只有转发作用的空模块和多层 `mod` 包装。模块树应该帮助定位，不应增加跳转成本。

## File Size And Function Size

- Rust 源文件建议控制在 150 到 300 行；超过 400 行时默认评估是否存在职责过多。
- 超过 600 行通常应视为拆分信号，除非该文件是明确的 schema、generated wrapper、或集中声明文件。
- 单个函数建议优先控制在 20 到 60 行；超过 80 行时，先检查是否能抽离独立决策块或重复逻辑。
- 不要为了压函数行数而抽出只有一行调用、反而破坏上下文的“伪抽象”。

## Readability

- 命名优先表达业务语义，不用模糊缩写，也不要堆砌过度学术化名称。
- 优先早返回、`let else`、`match` 和小范围局部变量，避免多层嵌套。
- `where` 子句优先用于承载较长 trait bound，保持函数签名可读。
- 公共边界和复杂局部逻辑应选择更直白的写法，避免炫技式宏、链式调用和隐式控制流。
- 克制使用 `pub`，优先私有，再通过构造函数、方法或受控 re-export 暴露稳定接口。

## Documentation And Comments

- crate 根和关键模块使用 `//!` 说明职责、边界、对外入口和重要限制。
- 所有公共类型、公共函数、公共 trait、公共宏都使用 `///` 文档注释。
- 公共方法文档默认包含：一句话摘要、参数或返回值说明、可能的错误、可能的 panic、必要时的示例。
- `unsafe fn`、`unsafe` block、并发共享状态、锁顺序、性能取舍、所有权边界、缓存或顺序依赖必须写注释，说明不变量与约束。
- 私有代码只注释“为什么”，不重复“代码已经写明白的是什么”。
- 容易引发分歧的决策点要留下说明，例如为何 clone、为何不用泛型、为何保留同步实现、为何选择某个 crate。

## Public API And Abstraction

- trait 只有在确实存在多个实现、清晰的扩展点、或稳定的测试替身需求时再引入。
- 默认不要为未来可能的需求预抽象；先写具体实现，等变化模式稳定后再提炼 trait 或泛型。
- 复杂配置优先使用配置结构体或 builder，不用长参数列表。
- 领域值优先考虑 newtype，避免裸 `String`、裸 `u64` 在公共边界上传播语义不清。
- feature flag 应映射到明确能力边界，不要用 flag 隐藏结构问题或制造相互纠缠的组合爆炸。

## Error Handling

- library 或公共模块优先返回结构化错误类型，如自定义 error enum 或 `thiserror`。
- 应用层、命令编排层可以在边界使用 `anyhow` 一类聚合错误，但不要让其污染底层领域逻辑。
- 生产代码默认避免 `unwrap` 和 `expect`；若确实依赖不可变式，错误信息必须明确说明假设。
- 遇到可恢复错误时返回 `Result`，不要静默吞错或仅记录日志后继续执行。

## Concurrency And Async

- 优先所有权转移、消息传递和不可变共享，其次才是 `Arc<Mutex<_>>` 这类共享可变状态。
- 在 async 代码中避免阻塞调用；若必须阻塞，明确隔离到专门边界并加注释。
- 共享状态、锁顺序、取消语义、超时和重试策略需要在实现附近注明。

## Unsafe

- 默认禁止 `unsafe`；只有在性能、FFI、底层内存布局等场景确有必要时才引入。
- `unsafe` 代码应尽量收缩到最小边界，并配套 `Safety` 文档说明调用方和实现方的不变量。
- 引入 `unsafe` 时必须补测试，并优先说明为什么安全替代方案不可行。

## Testing And Tooling

- 变更公共行为、边界条件、错误路径或修复 bug 时，必须补对应测试。
- 单元测试放在实现附近，跨模块或集成行为放在 `tests/`。
- 提交前至少执行 `cargo fmt`、`cargo clippy --all-targets --all-features -D warnings` 和相关测试。
- 不要全局关闭 lint；局部 `allow` 需要有简短理由。

## Dependencies

- 优先标准库和成熟、维护活跃、依赖面小的 crate。
- 新增依赖前先判断是缺能力，还是本地抽象设计不合理。
- 对宏密集、运行时绑定强、或体积较大的依赖要更谨慎，并在评审说明引入理由。

## Pending Project Decisions

- 是否默认要求 `#![forbid(unsafe_code)]`，还是只保留“默认禁止 `unsafe`”的指导原则。
- 是否对 `clippy` 采用零 warning 策略，还是允许带说明的局部例外。
- 是否为 async、FFI、workspace 多 crate 项目补充更细粒度的专项规则。
