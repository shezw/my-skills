## 场景

创建通用的 agent 编码闭环规范技能，并补齐默认生效的基线指令与本次任务记录。

### 开发目标

#### 1. 开发内容

- 新建可复用的 `agent-baseline` 技能，覆盖 `.agent` 目录、todo、tests、logs` 规范。
- 新建常驻基线指令，使编码任务默认遵守该规范。
- 为本次任务补齐 `req`、`res`、`tests`、`logs` 相关记录。

#### 2. 预期结果

- 形成一个可被发现和复用的通用规范技能。
- 形成一个默认生效的编码基线指令。
- `.agent` 目录下存在可追踪的任务、测试与日志闭环记录。

#### 3. 验证方式

- 检查新增技能与指令的 frontmatter 和内容是否完整。
- 初始化 `.agent/tests/results.db` 并写入本次任务的测试记录。
- 执行 git 状态检查，确认本次新增产物完整可见。

### 需求原文

## 1. 创建一个通用的基础规范技能，任何一次实际的编码行为都需要遵守。

主要内容如下：

### .agent文件目录与作用

在任意项目中，创建`.agent`文件夹，并维持 `req, res, todo, logs, tests` 子目录
- req: 开发者对agent提出的要求
- res: agent对于特定需求的回复与输入，如分析、建议、要点记录等
- todo: agent需要进行的任务，符合 agent-todo 技能规范要求
- logs: agent每次进行阶段性开发，问题修复等需要进行的日志记录
- tests: agent进行测试，无论成功与否，需要将测试内容进行记录

### 具体开发步骤

1. 如果需要，在res中回复 `res/res-{YMDHMin}-{question}.md`
2. 按照开发闭环目标规范，创建 `todo/task-{YMDHMin}.md`
3. 按照开发测试验证规范，创建 `tests/{sence}-{YMDHMin}-{target}.md`
4. 根据开发目标进行开发任务，并同步创建测试用例
5. 检查是否剩余问题未解决，未解决则继续开发
6. 检查git状态，提交代码，并记录提交日志 `logs/log-{YMDHMin}.md`


## 2. agent-todo 规范

1. 每次开发任务必须在 `todo/task-{YMDHMin}.md` 中记录开发目标、开发内容、预期结果等信息

格式为：

```markdown
## 场景(如有)

### 开发目标

#### 1.开发内容

- 具体功能点...

#### 2.预期结果

#### 3.验证方式
```

2. 开发目标需要明确具体，具有可操作性，并且能够通过测试验证来判断是否达成

3. 开发过程中需要符合相应的编码规范和最佳实践，确保代码质量和可维护性

4. 每次开发完成后，需要进行自我检查，确保开发内容符合预期结果，并且能够通过测试验证来判断是否达成开发目标

## 3. tests 开发闭环规范

1. tests 中使用SQLite建立 results.db 其格式为

```
id, project, scene, target, create_time, fails, passed_time
int-auto-increment, text, text, text, datetime, int, datetime
```

2. 创建测试用例时，必须在 `tests/{sence}-{YMDHMin}-{target}.md` 中记录测试内容，并同步在 `results.db` 中创建一条记录，记录测试的基本信息，如项目名称、测试场景、测试目标、创建时间等，并将 id 获取创建对应的测试用例文件名

3. 测试验证用例包含测试验证与结果判定，其中测试验证用于执行测试过程，结果判定用于闭环单个开发结果

> !!! **注意** 用例始终返回一个布尔值，用例程序内部逻辑判断分支决定测试是否通过，并使用对应ID，将测试结果需要记录在 `results.db` 中，失败增加 fails 字段的值，成功则记录 passed_time 字段的时间戳

4. 每次开发完成后，必须执行对应的测试用例进行验证，并根据测试结果进行问题修复或记录成功日志

## 4. logs 日志记录规范

1. 每次开发阶段性完成后，必须在 `logs/log-{YMDHMin}.md` 中记录开发日志，包括开发内容、遇到的问题、解决方案等

2. 日志记录需要详细且清晰，以便后续回顾和分析开发过程中的问题和解决方案
