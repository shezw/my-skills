# React Review Checklist

## Structure

- 模块是否按 feature 拆分，而不是横向技术分类。
- 模块深度是否保持在默认 3 层以内。
- 页面是否只负责页面组装和数据边界，而不是把所有业务逻辑塞在一个文件里。
- 是否避免了无意义 barrel、纯转发组件和过度碎片化拆分。

## Components And Hooks

- 组件职责是否单一，是否一个文件只有一个主要组件。
- 展示组件是否尽量纯粹，避免直接耦合数据获取。
- 自定义 hook 是否真的有复用价值或复杂状态价值，而不是为了“拆分而拆分”。
- 是否避免把可推导值重复存成 state。

## Readability

- 命名是否具有业务语义。
- 是否使用早返回、清晰分支和稳定的 JSX 结构。
- 是否避免了过大的匿名函数、链式三元表达式和难懂的 memo 化。
- 必要的注释是否解释了“为什么”，而不是重复“是什么”。

## UI And Styling

- 是否优先采用 Mantine 现成能力而不是重复造基础 UI。
- 图标是否统一使用 `@tabler/icons-react`。
- 颜色、间距、字体等是否走 theme，而不是硬编码。
- 是否避免混用多个 UI、表单或弹层体系。

## Data, Forms, And Async

- 数据获取是否位于页面边界、feature hook 或 `api` 模块。
- 加载、空态、错误态是否是显式 UI 分支。
- 表单是否默认使用 Mantine form 和结构化校验。
- effect 是否只用于与外部系统同步，而不是做派生计算。

## Accessibility And Testing

- 交互控件是否支持键盘访问，焦点态是否保留。
- 图标按钮、表单控件和状态反馈是否具备可访问名称和可理解文案。
- 是否为新增行为、错误态、权限分支和回归问题补了测试。
- 测试是否关注用户行为，而不是内部实现细节。

## Performance And Dependencies

- 是否存在可以更早拆分的路由或重组件边界。
- 是否避免了不必要的全局状态和无证据 memo 化。
- 是否优先使用成熟优质库，而不是新增重复能力依赖。
- 是否计划执行 lint、类型检查和相关测试。
